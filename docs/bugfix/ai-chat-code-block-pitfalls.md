# 代码块渲染踩坑记录

## 问题 1：代码块内换行被替换为 `<br>` 标签

### 现象

AI 回复中的代码块显示为：

```cpp
// C++ 示例代码<br>#include <iostream><br>#include <vector><br><br>int fibonacci(int n) {<br>    if (n <= 1) return n;<br>
```

`<br>` 被原样展示，而非渲染为换行。代码内容完全无法阅读。

### 根因

`renderMarkdownLite` 的处理顺序是：

```
步骤 1: 匹配 ``` 代码块，渲染为 <div class="msg-code-block">...</div>
步骤 2: 处理行内代码、加粗、列表
步骤 3: 将所有 \n 替换为 <br>
```

步骤 3 的 `\n → <br>` 是全局替换，会破坏已渲染的代码块 HTML 内部结构。例如步骤 1 生成的 `<pre>line1\nline2\n</pre>` 经过步骤 3 变成 `<pre>line1<br>line2<br></pre>`——`<pre>` 内不该有 `<br>`，HTML `<pre>` 本身保留换行。

此外，当代码块在流式输出中尚未闭合（只有开头的 ` ``` ` 没有结尾的 ` ``` `），步骤 1 的正则 `/```(\w+)?\n([\s\S]*?)```/g` 无法匹配，整个代码块文本被当作普通段落处理，所有 `\n` 全部变成 `<br>`。

### 修复

改为**提取-保护-还原**模式，用不可见占位符 `\x00CODEBLOCK\x00` 暂存代码块内容：

```
步骤 1: 提取所有代码块（含未闭合的），用占位符替换，防止内容被后续规则破坏
步骤 2: 提取行内代码，用占位符替换
步骤 3: 处理加粗、列表、\n → <br>（此时代码块和行内代码已被保护）
步骤 4: 还原行内代码，渲染为 <code>...</code>
步骤 5: 还原代码块，渲染为 <div class="msg-code-block">...</div>
```

未闭合代码块的正则改为 `/```(\w+)?\n([\s\S]*?)(?:```|$)/g`，`$` 分支匹配到文本末尾，流式中未闭合的代码块也能被提取并渲染为代码块样式。

### 教训

**Markdown 渲染必须先提取代码块再处理其余语法**。代码块内的内容是"纯文本"，不应受任何后续规则影响。这是所有 Markdown 解析器（marked、markdown-it 等）的基本设计原则——保护代码块内容不被误处理。

占位符用 `\x00`（null byte）而非常见字符串（如 `{{CODE}}`），因为 `\x00` 不可能在用户文本中出现，避免了占位符被误替换的风险。

## 问题 2：流式输出中未闭合代码块渲染为普通文本

### 现象

模型流式输出代码时，先发出 ` ```cpp\n`，然后逐 token 发出代码内容，最后才发出 ` ``` `。在代码内容还没完全输出时，中间的文本段包含 ` ```cpp\n#include...` 但没有闭合的 ` ``` `，整个代码段被渲染为普通文本（灰色、无语法高亮），看起来像一堆乱码。

### 根因

原正则 `/```(\w+)?\n([\s\S]*?)```/g` 要求代码块必须有闭合的 ` ``` ` 才能匹配。流式输出中：

```
token 1: "```cpp\n"           → 无闭合，不匹配
token 2: "#include <iostream>" → 无闭合，不匹配
token 3: "\nint main() {"     → 无闭合，不匹配
...
token N: "```"                 → 此时闭合，才能匹配
```

在 token 1 到 token N-1 期间，代码块文本被当作普通段落处理：
- `\n` 全部变成 `<br>`
- `#include <iostream>` 中的 `<iostream>` 被 `escapeHtml` 转义为 `&lt;iostream&gt;`
- 代码关键字没有语法高亮

用户看到的就是一段带 `<br>` 和转义标签的乱码文本，直到流式完成后整个消息重新渲染才变成正确的代码块。

### 修复

正则改为 `/```(\w+)?\n([\s\S]*?)(?:```|$)/g`：
- `(?:```|$)` — 匹配闭合的 ` ``` ` 或文本末尾 `$`
- `$` 分支让未闭合的代码块也能被提取，渲染为代码块样式

虽然 `$` 分支的代码块没有结尾的 ` ``` `，但代码块 HTML 本身就是一个完整的 `<div class="msg-code-block">` 容器，视觉上就是代码块样式。随着后续 token 到达，`renderAllMessages()` 会重新渲染，最终闭合的 ` ``` ` 被匹配，代码块从"未闭合版本"更新为"完整版本"。

### 教训

**流式场景下的渲染必须是增量兼容的**。不能假设文本已经完整——每个中间状态都应该有合理的视觉呈现。对于代码块，"未闭合但渲染为代码块样式"远好于"未闭合就渲染为普通文本"。

## 问题 3：代码块配色异常（看不清内容）

### 现象

代码块内容颜色很淡或看不清，像被一层灰色覆盖。

### 根因

两层 CSS 问题叠加：

1. `.msg-code-content` 设了 `color: #d4d4d4`，这个属性会通过 CSS 继承传递到所有子元素。Monaco `colorize` 用 `mtk*` class（如 `mtk1`, `mtk3`）给 `<span>` 设置具体颜色，但如果 `.msg-code-content` 的 `color` 在 CSS 优先级上高于 `mtk*` 的 `color`，就会覆盖 Monaco 的主题颜色。

2. Monaco `colorize` 默认使用编辑器当前主题。如果编辑器用的是 `vs`（light 主题），colorize 就生成浅色配色（白色/浅灰文字），放到深色背景 `#1e1e1e` 的代码块里，浅色文字配深色背景本身应该没问题，但 `vs` 主题的某些语法颜色（如注释的 `#008000` 绿色）在深色背景下不够醒目，而 `vs-dark` 主题的颜色（如注释的 `#6A9955`）更适合深色背景。

### 修复

- `.msg-code-content` 移除 `color` 属性，只保留 `font-family` 等布局属性，让 Monaco `mtk*` span 的颜色完全生效
- fallback 颜色只在 `.msg-code-content pre` 上设置（colorize 未执行时）
- `renderCodeBlocksAsync` 强制传入 `{ theme: 'vs-dark' }`，因为代码块背景始终是 `#1e1e1e`，无论 UI 整体主题是 light 还是 dark

### 教训

**代码块是独立的视觉区域，配色应与自身背景匹配，而非跟随全局主题**。深色背景的代码块必须用深色主题的语法高亮颜色。全局 `color` 属性通过 CSS 继承可能意外覆盖第三方库的颜色设置，对第三方渲染结果不应设全局 `color`。
